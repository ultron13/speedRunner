## Phase 1: Refactor State Management to Redis

### 1.1 Create `frontend/lib/redis.ts`
- Redis client using `ioredis`
- Connection pooling with configurable URL from env
- Graceful connection handling (retry, reconnect)

### 1.2 Create `frontend/lib/state.ts`
Abstraction layer over Redis for state operations:
```typescript
// Keys stored in Redis:
// speedrunner:tests       → JSON array of Test[]
// speedrunner:runs        → JSON array of Run[]
// speedrunner:liveMetrics → JSON object { [testId]: LiveMetrics }
// speedrunner:trendData   → JSON array of TrendPoint[]

// Functions:
getTests() / setTests(tests)
getRuns() / setRuns(runs)
getLiveMetrics() / setLiveMetrics(metrics)
getTrendData() / setTrendData(trendData)
getSnapshot() → full state object
hydrate() → seed initial mock data if Redis is empty
```

### 1.3 Refactor `frontend/server.ts` and `frontend/server.production.ts`
- Replace all in-memory variable access with `state.*` calls
- Remove `let tests`, `let runs`, etc. module-level variables
- Import and initialize Redis connection on startup
- Add Redis connection health to the health check endpoint
- Close Redis connection on shutdown (SIGTERM/SIGINT handlers)

### 1.4 Add `ioredis` dependency
- `npm install ioredis`
- `npm install -D @types/ioredis` (if needed)

### 1.5 Update mock data seeding
- `hydrate()` checks if Redis has data; if empty, seeds with `createMockData()`
- If data exists, uses existing Redis state (survives pod restarts)

## Phase 2: Fix & Harden Docker Build

### 2.1 Rewrite `Dockerfile`

**Problems to fix:**
- `npm ci --only=production` → `npm ci --omit=dev`
- Standalone output needs custom server entrypoint
- Health check uses `require('http')` → use `wget` (available in alpine)
- Add proper OCI labels

**New structure:**
```
Stage 1 (deps):      npm ci --omit=dev
Stage 2 (builder):   COPY source, npm run build (includes server.production.ts)
Stage 3 (runner):    Copy standalone + static + public, non-root user, wget healthcheck
```

### 2.2 Update `Dockerfile.dev`
- Add `CHOKIDAR_USEPOLLING=true` for Docker volume watch
- Expose all 3 ports (8787, 8788, 9090)
- Install all deps (including devDependencies)

### 2.3 Update `.dockerignore`
- Add `.mimocode/`, `k8s/`

### 4.1 `Chart.yaml`
- name: speedrunner
- version: 0.1.0
- appVersion: 0.1.0
- dependencies: redis (bitnami chart, optional)

### 4.2 `values.yaml` (defaults)
```yaml
replicaCount: 2

image:
  repository: ghcr.io/<owner>/speedrunner
  tag: "latest"
  pullPolicy: IfNotPresent

service:
  type: ClusterIP
  httpPort: 8787
  wsPort: 8788
  healthPort: 9090

ingress:
  enabled: true
  className: nginx
  annotations:
    cert-manager.io/cluster-issuer: letsencrypt-prod
  hosts:
    - host: speedrunner.example.com
      paths:
        - path: /
          pathType: Prefix
  tls:
    - secretName: speedrunner-tls
      hosts:
        - speedrunner.example.com

redis:
  enabled: true
  host: ""
  port: 6379

postgres:
  enabled: true
  host: ""
  port: 5432
  database: speedrunner
  user: speedrunner

resources:
  requests:
    cpu: 250m
    memory: 256Mi
  limits:
    cpu: 500m
    memory: 512Mi

autoscaling:
  enabled: true
  minReplicas: 2
  maxReplicas: 10
  targetCPUUtilizationPercentage: 70
  targetMemoryUtilizationPercentage: 80

podDisruptionBudget:
  enabled: true
  minAvailable: 1

env:
  NODE_ENV: production
  LOG_LEVEL: info
  LOG_FORMAT: json
```

### 4.4 `templates/app-deployment.yaml`
- Configurable replicas from values
- Image from values
- Resource limits from values
- Liveness probe: HTTP GET `/health` port 9090
- Readiness probe: HTTP GET `/health` port 9090
- Startup probe: HTTP GET `/health` port 9090, failureThreshold 30
- Rolling update: maxSurge 1, maxUnavailable 0
- Security context: non-root (1001), readOnlyRootFilesystem, drop all caps
- Environment from ConfigMap + Secrets
- Redis/PostgreSQL connection info from values

### 4.5 `templates/ingress.yaml`

- Nginx ingress class

- WebSocket upgrade annotations for `/ws` path
- Rate limiting annotations

### 4.6 `templates/app-hpa.yaml`
- Conditional on `autoscaling.enabled`
- Min/max replicas from values
- CPU/memory targets from values

### 4.7 Other templates
- Standard Kubernetes resource definitions with Helm templating
- All using `{{ include "speedrunner.fullname" . }}` naming convention
- Proper labels via `{{ include "speedrunner.labels" . }}`

## Phase 7: Verification

### Redis state verification:
1. Start app with Redis running
2. Create a test via WebSocket → verify it persists in Redis
3. Stop and restart app → verify test data survives
4. Check Redis directly: `redis-cli KEYS "speedrunner:*"`

### Docker verification:
1. `docker-compose build` — image builds
2. `docker-compose up` — all services start, health checks pass
3. `curl http://localhost:9090/health` — returns JSON with Redis connected
4. WebSocket connection to `ws://localhost:8788` — connects, receives snapshot

### Kubernetes verification:
1. `helm install speedrunner ./helm/speedrunner -n speedrunner --create-namespace`
2. `kubectl get pods -n speedrunner` — all pods running
3. `kubectl get svc -n speedrunner` — all services created
4. `kubectl get ingress -n speedrunner` — ingress configured
5. `kubectl get hpa -n speedrunner` — autoscaler active
6. `kubectl port-forward svc/speedrunner 8787:8787 -n speedrunner` — app accessible
7. Health check responds with Redis status

### Regression verification:
1. `npm run test:run` — all unit tests pass
2. `npm run typecheck` — zero type errors
3. `npm run lint` — zero lint errors
4. `npx playwright test` — all e2e tests pass
5. `npm run build` — production build succeeds

3. **Phase 3** (Docker Compose) — depends on Phase 2
4. **Phase 4** (Helm chart) — can be done in parallel with Phase 3
5. **Phase 5** (Makefile) — after Phases 3-4
6. **Phase 6** (nginx) — after Phase 3
7. **Phase 7** (Verification) — after all phases
