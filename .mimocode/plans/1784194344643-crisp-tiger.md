# SpeedRunner Enterprise — Next Implementation Phase

## Context

SpeedRunner Enterprise has an **extensive frontend** (85+ components, 18 Zustand stores, 1113-line type system, Docker/Helm/K8s deployment) but **no real backend** beyond a WebSocket mock server with simulated metrics. The `Implementation/` folder contains 32 detailed design documents (700+ page backlog) for a LoadRunner Enterprise-class platform.

The OpenPE design document and the existing backlog both converge on the same priority: **build the backend services that connect the polished UI to real data and real load generation**.

## What Exists vs. What's Needed

| Layer | Current State | Gap |
|---|---|---|
| **Frontend UI** | 85+ React components, 18 stores, full dashboard | Complete — no work needed |
| **REST API** | 1 health endpoint (`app/api/health/route.ts`) | Needs full CRUD for tests, runs, users, schedules, reports |
| **Database** | PostgreSQL in Helm/docker-compose but never connected | Needs schema, migrations, ORM |
| **State Layer** | Redis KV with in-memory fallback (`lib/state.ts`) | Partially works — needs Postgres for durable data |
| **Load Generation** | Simulated via `lib/simulation.ts` random-walk | Needs real HTTP load generation |
| **WebSocket** | Mock server broadcasts fake metrics (`server.production.ts`) | Needs to broadcast real metrics from actual test runs |
| **Auth** | Hardcoded users, JWT stubs | Needs real auth with Postgres-backed users |
| **Messaging** | None | Needs event bus for async communication between services |
| **Object Storage** | None | Needs MinIO/S3 for artifacts (logs, reports, scripts) |

## Recommended Next Phase: REST API + PostgreSQL + Real Test Execution

This is the **Phase 1** from both the OpenPE roadmap and the `32-Full-Project-Implementation-Backlog.md`. It transforms the platform from a UI prototype into a working system.

### Deliverables (5 work streams)

---

### WS-1: PostgreSQL Schema + Prisma ORM

**Why first**: Every other service needs durable storage. Without it, nothing persists.

**Files to create:**
- `frontend/prisma/schema.prisma` — Complete data model
- `frontend/lib/db.ts` — Prisma client singleton
- `frontend/prisma/seed.ts` — Seed data (migrate from `data/mock-data.ts`)

**Schema tables (derived from `types/index.ts`):**
- `users` — id, email, name, passwordHash, role, createdAt
- `projects` — id, name, description, ownerId, createdAt
- `tests` — id, projectId, name, description, scriptType, targetUrl, virtualUsers, status, createdAt
- `runs` — id, testId, status, startedAt, completedAt, duration, throughput, avgResponseTime, p50, p90, p95, p99, errorRate, triggeredBy
- `run_metrics` — id, runId, timestamp, throughput, avgResponseTime, errorRate, activeVUsers (time-series per-run)
- `sla_thresholds` — id, projectId, name, metric, condition, value, enabled
- `sla_violations` — id, runId, thresholdId, actualValue, passed
- `test_templates` — id, projectId, name, description, scriptType, targetUrl, virtualUsers, usageCount
- `test_schedules` — id, testId, frequency, cronExpression, enabled, nextRunAt
- `audit_logs` — id, userId, action, resourceType, resourceId, details, timestamp
- `api_keys` — id, userId, name, keyHash, lastUsedAt, expiresAt

**Migration strategy:** Use `prisma migrate dev` for development, `prisma migrate deploy` in CI/CD.

### WS-2: REST API Layer

**Why**: The frontend has 85+ components that currently talk to a mock WebSocket. They need real endpoints.

**Files to create:**
- `frontend/app/api/tests/route.ts` — GET (list), POST (create)
- `frontend/app/api/tests/[id]/route.ts` — GET, PUT, DELETE
- `frontend/app/api/tests/[id]/start/route.ts` — POST (start test)
- `frontend/app/api/tests/[id]/stop/route.ts` — POST (stop test)
- `frontend/app/api/runs/route.ts` — GET (list with filters)
- `frontend/app/api/runs/[id]/route.ts` — GET (single run details)
- `frontend/app/api/runs/[id]/metrics/route.ts` — GET (time-series metrics)
- `frontend/app/api/schedules/route.ts` — CRUD
- `frontend/app/api/sla/route.ts` — CRUD
- `frontend/app/api/templates/route.ts` — CRUD
- `frontend/app/api/auth/login/route.ts` — POST (login)
- `frontend/app/api/auth/me/route.ts` — GET (current user)
- `frontend/app/api/audit/route.ts` — GET (audit log)

**Pattern:** Each route file imports Prisma client, validates input, queries DB, returns JSON. Use Next.js App Router conventions.

**Auth middleware:** Extend `middleware.ts` to validate JWT on all `/api/*` routes except `/api/auth/login` and `/api/health`.

### WS-3: Real Load Generator Service

**Why**: The core value of the platform. Currently `lib/simulation.ts` fakes metrics with random walks.

**Approach:** Build a lightweight HTTP load generator in Node.js (not Go — stay in the existing stack for v1). Use `undici` (built into Node 18+) for high-performance HTTP requests.

**Files to create:**
- `frontend/lib/load-generator.ts` — Core load generation engine
  - `generateLoad(config)` — Runs concurrent HTTP requests against target URL
  - Configurable: virtual users, duration, ramp-up, think time
  - Emits metrics events (throughput, latency, errors) in real-time
  - Reports results back to the store via callback
- `frontend/lib/load-generator-worker.ts` — Worker thread for CPU-bound metric aggregation

**Integration with existing store:**
- When `startTest(testId)` is called, instead of `tick()` random walk, actually fire HTTP requests
- Real metrics flow through the same WebSocket broadcast path
- `lib/simulation.ts` becomes a fallback for demo/offline mode

**Key design decisions:**
- Use Node.js worker_threads to avoid blocking the event loop
- Each test run gets its own worker instance
- Workers communicate metrics back via postMessage
- Configurable concurrency (default: cap at 1000 VUsers per pod)

---

### WS-4: WebSocket Server Enhancement

**Why**: The existing WebSocket server (`server.production.ts`) broadcasts fake ticks. It needs to consume real metrics from the load generator.

**Files to modify:**
- `frontend/server.production.ts` — Replace `tick()` interval with real metrics consumption
- `frontend/lib/state.ts` — Add `publishMetrics(runId, metrics)` method

**Changes:**
- When a test starts, spawn a load generator worker
- Worker emits metrics events → state store updates → WebSocket broadcasts to connected clients
- Maintain backward compatibility: if load generator unavailable, fall back to simulation mode

---

### WS-5: Frontend API Integration

**Why**: The 85+ frontend components currently use Zustand stores with mock data. They need to fetch from real API endpoints.

**Files to modify:**
- `frontend/store/test-store.ts` — Add `fetchTests()`, `fetchRuns()` that call REST API
- `frontend/store/auth-store.ts` — Connect to `/api/auth/login` instead of hardcoded users
- `frontend/store/reporting-store.ts` — Fetch templates from API
- `frontend/store/alerting-store.ts` — Fetch SLA thresholds from API

**Pattern:** Add async actions to stores that:
1. Call REST API
2. Update Zustand state
3. Fall back to local state if API unavailable (offline mode)

---

## Implementation Order

```
WS-1 (DB Schema)  →  WS-2 (REST API)  →  WS-5 (Frontend Integration)
                                            ↓
WS-3 (Load Generator)  →  WS-4 (WebSocket Enhancement)
```

1. **WS-1 first** — DB schema is the foundation everything else depends on
2. **WS-2 parallel with WS-3** — API layer and load generator are independent
3. **WS-4 after WS-3** — WebSocket needs real metrics from load generator
4. **WS-5 last** — Frontend integration connects everything

## Verification Plan

1. **Database**: `npx prisma migrate dev` creates all tables, `npx prisma db seed` populates with mock data
2. **API**: `curl localhost:8787/api/tests` returns JSON array of tests from PostgreSQL
3. **Auth**: `curl -X POST localhost:8787/api/auth/login -d '{"email":"admin@speedrunner.dev","password":"..."}'` returns JWT
4. **Load Generation**: Start a test via API → real HTTP requests fire against target → metrics stream via WebSocket → dashboard shows live data
5. **End-to-end**: Login → create test → start test → watch live metrics → stop test → view results → generate report
6. **TypeScript**: `npx tsc --noEmit` passes
7. **Tests**: `npx vitest run` — all existing 166 tests pass + new API/store tests
8. **Build**: `npx next build` succeeds
9. **Docker**: `docker compose up` starts all services (app + postgres + redis)
