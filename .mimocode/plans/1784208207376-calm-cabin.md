# MarathonRunner Enterprise — Full Implementation Plan

## Current State Assessment

**What exists today:**
- **Next.js frontend** (`frontend/`) — 90+ React components, Zustand stores, WebSocket real-time metrics, Prisma schema (PostgreSQL), API routes for tests/runs/auth/audit/SLA/templates
- **Simulation engine** (`lib/simulation.ts`) — generates fake metrics via random walk; no real test execution
- **Infrastructure** — Docker Compose (app + Redis + PostgreSQL + Nginx), Helm chart (18 templates), GitHub Actions CI/CD (5 workflows)
- **Auth** — basic JWT with bcrypt password hashing; no OIDC/SAML
- **Database** — Prisma schema covers users, projects, tests, runs, metrics, SLA, templates, schedules, audit, API keys

**What the Implementation docs describe (the full vision):**
- 15+ microservices with messaging, K8s Operator, KEDA autoscaling
- Real JMeter/k6/Gatling/Locust/Playwright execution on Kubernetes
- Jenkins pipeline orchestration, Ansible provisioning, Rancher management, Harbor registry
- OIDC/SAML auth, full RBAC, policy-as-code
- AI features (anomaly detection, bottleneck correlation, test design assistant, summaries)
- Multi-region, disaster recovery, object storage, monitoring stack

**Architecture decision:** Hybrid — Next.js frontend + Go backend services + Kubernetes execution from day one.

**Dual-mode development:**
- **Docker Compose** — fast iteration on backend/frontend/API without K8s overhead. Use for auth, API routes, database, UI development.
- **Minikube** — full K8s-native execution testing. Use when testing JMeter pod orchestration, Helm deployments, RBAC, network policies, and execution workflows.
- Both modes share the same Go backend code — `KUBECONFIG` env var controls whether to use in-cluster or local K8s client.

## Prerequisite: Local Kubernetes Environment

**Available tools:** minikube (installed), kubectl (installed), docker 29.1.3
**Missing tools:** helm, kind, k3d, k3s

### Option A: Minikube (Recommended — already installed)

Minikube is the simplest path since it's already on the machine.

**Setup steps:**
1. Start minikube with sufficient resources for JMeter execution:
   ```bash
   minikube start --cpus=4 --memory=8192 --disk-size=40g --driver=docker
   ```
2. Enable required addons:
   ```bash
   minikube addons enable ingress        # NGINX ingress controller
   minikube addons enable metrics-server # resource metrics for HPA
   minikube addons enable storage-provisioner
   ```
3. Configure Docker to use minikube's Docker daemon (avoids image pull):
   ```bash
   eval $(minikube docker-env)
   # Build images directly into minikube's Docker
   ```
4. Install Helm:
   ```bash
   curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash
   ```
5. Create namespaces:
   ```bash
   kubectl create namespace marathonrunner-system
   kubectl create namespace marathonrunner-execution
   kubectl create namespace marathonrunner-observability
   ```

**Resource plan for minikube (4 CPU / 8GB):**
| Component | CPU Request | Memory Request | Notes |
|-----------|------------|----------------|-------|
| Go backend | 250m | 256Mi | Platform control plane |
| Next.js frontend | 250m | 256Mi | Portal UI |
| PostgreSQL | 250m | 256Mi | Metadata store |
| Redis | 100m | 128Mi | Runtime state |
| MinIO | 100m | 128Mi | Object storage |
| Nginx | 100m | 128Mi | Reverse proxy |
| JMeter coordinator | 500m | 512Mi | Per-run, transient |
| JMeter workers (x2) | 500m each | 512Mi each | Per-run, transient |
| **Total** | **~2.1 CPU** | **~2.3 Gi** | Leaves headroom for JMeter |

**Limitations:** Single-node cluster, no real multi-node scheduling, no persistent volumes for production use. Sufficient for development and testing.

### Setup Script: `k8s/scripts/setup-local-cluster.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail

echo "=== Setting up local Kubernetes cluster ==="

# 1. Start minikube
minikube status | grep -q "Running" || \
  minikube start --cpus=4 --memory=8192 --disk-size=40g --driver=docker

# 2. Enable addons
minikube addons enable ingress
minikube addons enable metrics-server

# 3. Install Helm (if missing)
command -v helm >/dev/null || \
  curl https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3 | bash

# 4. Create namespaces
kubectl create namespace marathonrunner-system --dry-run=client -o yaml | kubectl apply -f -
kubectl create namespace marathonrunner-execution --dry-run=client -o yaml | kubectl apply -f -
kubectl create namespace marathonrunner-observability --dry-run=client -o yaml | kubectl apply -f -

# 5. Apply local storage
kubectl apply -f k8s/postgres/local-pvc.yaml
kubectl apply -f k8s/redis/local-pvc.yaml
kubectl apply -f k8s/minio/local-pvc.yaml

# 6. Build images into minikube Docker
eval $(minikube docker-env)
docker build -t speedrunner-backend:latest -f backend/Dockerfile backend/
docker build -t speedrunner-frontend:latest -f Dockerfile frontend/

echo "=== Local cluster ready ==="
minikube dashboard --url &
```

### Deploy Script: `k8s/scripts/deploy-platform.sh`

```bash
#!/usr/bin/env bash
set -euo pipefail

NAMESPACE="marathonrunner-system"

echo "=== Deploying MarathonRunner to local K8s ==="

# 1. Build images
eval $(minikube docker-env)
docker build -t speedrunner-backend:latest -f backend/Dockerfile backend/
docker build -t speedrunner-frontend:latest -f Dockerfile frontend/

# 2. Install Helm chart
helm upgrade --install speedrunner helm/speedrunner \
  -n $NAMESPACE \
  -f k8s/local-values.yaml \
  --set image.repository=speedrunner-frontend \
  --set image.tag=latest \
  --set backend.image.repository=speedrunner-backend \
  --set backend.image.tag=latest \
  --wait --timeout 5m

# 3. Verify
kubectl get pods -n $NAMESPACE
echo "=== Platform deployed ==="
echo "Frontend: $(minikube service speedrunner -n $NAMESPACE --url 2>/dev/null)/"
```

### JMeter Container Image

Create `k8s/jmeter/jmeter-image/Dockerfile`:

```dockerfile
FROM apache/jmeter:5.6.3

# Install additional plugins
RUN wget -O /opt/jmeter/lib/ext/jmeter-plugins-manager.jar \
  https://jmeter-plugins.org/downloads/latest/jmeter-plugins-manager.jar

# Copy entrypoint
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh

ENTRYPOINT ["/entrypoint.sh"]
```

`k8s/jmeter/jmeter-image/entrypoint.sh`:
```bash
#!/bin/bash
set -e

# Environment variables injected by controller:
# - TEST_PLAN_URL or TEST_PLAN_PATH
# - TARGET_URL
# - VIRTUAL_USERS
# - DURATION
# - RAMP_UP
# - REDIS_URL
# - RUN_ID
# - RESULTS_OUTPUT

echo "Starting JMeter execution..."
echo "Run ID: ${RUN_ID}"
echo "Target: ${TARGET_URL}"
echo "VUsers: ${VIRTUAL_USERS}"
echo "Duration: ${DURATION}s"

exec /opt/jmeter/bin/jmeter \
  -n \
  -t "${TEST_PLAN_PATH:-/test-plan/test-plan.jmx}" \
  -Jtarget_host="${TARGET_URL}" \
  -Jthreads="${VIRTUAL_USERS}" \
  -Jramp_up="${RAMP_UP:-30}" \
  -Jduration="${DURATION}" \
  -l "/results/${RUN_ID}/results.jtl" \
  -j "/results/${RUN_ID}/jmeter.log"
```

**Metrics to compute:**

## Risk Mitigation

1. **K8s cluster availability** — docker-compose fallback for local development
2. **JMeter image size** — use official apache/jmeter image, cache in Harbor
3. **Result parsing complexity** — start with JTL XML, add CSV later
4. **AI costs** — make LLM provider configurable, support local models
5. **Scope creep** — strictly follow sprint order, defer lower-priority items
